Zephyr HIL Testbench Project Plan

A practical plan for building a mini Hardware-in-the-Loop testing system using Robot Framework, Python, Zephyr RTOS, ESP32 as the testbench, and STM32 as the DUT.

Robot Framework Python Zephyr RTOS ESP32 DAC STM32 ADC SPI / UART / PWM / GPIO

1. Project Goal

The goal is to build a mini HIL testbench that can automatically test embedded firmware running on a DUT board. The final target architecture is: 

PC running Robot Framework
        |
        | USB Serial
        v
ESP32 running Zephyr = Testbench Firmware
        |
        | GPIO / UART / SPI / PWM / DAC analog signal
        v
STM32 running Zephyr = DUT Firmware

The ESP32 testbench receives test commands from the PC, generates electrical signals, communicates with the STM32 DUT, reads responses, then sends results back to Robot Framework. Robot Framework decides whether each test case is PASS or FAIL.

Important: The project is not mainly about making an STM32 application. The project is about making a reusable embedded testing platform.

2. Current Situation and Temporary Plan

Right now you only have one STM32 board. That is okay. Start with a temporary loopback setup to build the project core before buying the ESP32.

Temporary Setup

PC Robot Framework
        |
        | USB Serial
        v
STM32 running Zephyr
        |
        | Loopback wires
        v
Same STM32 pins

This stage proves the command protocol, Robot integration, GPIO, UART, PWM, SPI basics, and project structure.

Final Setup

PC Robot Framework
        |
        | USB Serial
        v
ESP32 Testbench using Zephyr
        |
        | Real signals
        v
STM32 DUT using Zephyr

After the ESP32 arrives, move the testbench firmware from STM32 to ESP32 and keep STM32 as the real DUT.

Do not wait for ESP32 to start. Start now with STM32 loopback. The hard parts are the protocol, Zephyr drivers, Robot tests, test cases, and documentation — not only the board migration.

3. Final Deliverables

Deliverable Description
ESP32 Testbench Firmware Zephyr firmware that receives commands from PC and controls GPIO, UART, SPI, PWM, and DAC.
STM32 DUT Firmware Zephyr firmware that behaves like a small ECU and reacts to testbench inputs.
Robot Framework Test Suite Automated test cases for GPIO, UART, PWM, ADC/DAC, and SPI.
Python Serial Library Robot custom library that sends commands to the testbench and parses responses.
Test Reports Robot output: log.html, report.html, and output.xml.
Oscilloscope Evidence PWM and analog signal screenshots.
Saleae Evidence UART and SPI decoded captures.
Documentation Architecture, protocol, wiring, test plan, known limitations, and demo steps.
Presentation Demo A clear story showing how an automated test moves from Robot to hardware and returns PASS/FAIL.

5. Communication Protocol Between PC and Testbench

Use a simple text-based protocol first. It is easier to debug with a serial terminal, oscilloscope, Saleae, and Robot Framework.

Command Meaning Expected Response
PING Check if testbench is alive. OK PONG
GPIO_SET OUT1 1 Set testbench output pin HIGH. OK
GPIO_READ IN1 Read testbench input pin. OK 1 or OK 0
UART_SEND 55 AA Send UART frame from testbench to DUT. OK
UART_EXPECT CC 100 Wait up to 100 ms for DUT UART response 0xCC. OK MATCH or ERR TIMEOUT
PWM_SET CH1 10000 50 Generate 10 kHz PWM with 50% duty. OK
DAC_SET CH1 1500 Generate analog output of 1500 mV from ESP32 DAC. OK
ADC_READ CH1 Read analog voltage from DUT or loopback input. OK 1485
SPI_TXRX A5 5A Send SPI bytes and read DUT response. OK 12 34
Use logical names like OUT1, IN1, CH1 instead of real pin names in Robot. Keep pin mapping inside the firmware or documentation. This makes board migration easier.

6. Team Working Model

Since both team members want to understand the whole project, do not divide by layer. Do not make one person only Robot and the other only firmware. Instead, divide by feature. 

Feature Owner Model:

One person owns one complete feature from end to end:
Robot Test Case -> Python Library -> Testbench Command -> DUT Behavior -> Hardware Evidence -> Documentation
Feature Owner Reviewer Scope
GPIO Person A Person B Robot + Python + Testbench GPIO + DUT GPIO + test report.
UART Person B Person A Robot + Python + UART frame test + DUT response + Saleae capture.
PWM Person A Person B Robot + Python + PWM generation/check + oscilloscope evidence.
ADC/DAC Person B Person A ESP32 DAC output + STM32 ADC read + tolerance-based Robot test.
SPI Person A Person B Robot + Python + SPI command/response + Saleae SPI decode.
Rule: A feature is not done until the reviewer runs it, understands it, changes a small part, and approves the documentation.

7. Project Phases and Timeline

A realistic timeline is 18 to 24 days if both members are still learning Zephyr, Robot Framework, and embedded testing concepts.

Phase Duration Goal Output

Phase 1: Core Protocol 2 days PC sends command to board and receives response. PING -> OK PONG.
Phase 2: GPIO MVP 3 days First complete end-to-end test. Robot GPIO test PASS/FAIL report.
Phase 3: UART Feature 3 days Test DUT behavior when receiving a specific UART frame. UART Robot tests + Saleae decode.

Phase 5: SPI Feature 3 days Test SPI command/response behavior. SPI Robot test + Saleae decode.

8. Strong Test Cases for Presentation

GPIO Test Case: Safety Interlock

Story: The DUT should enable an output only when an enable input is HIGH.

Test Steps:
1. Robot sends GPIO_SET ENABLE 0
2. Testbench drives DUT input LOW
3. Robot reads DUT output
4. Expected output = 0
5. Robot sends GPIO_SET ENABLE 1
6. Expected output = 1

Good presentation point: This shows automated digital input/output validation.

UART Test Case: Valid Frame ACK

Story: The DUT waits for a specific UART command. If it receives the correct frame, it replies ACK.

Valid Frame: 0x55 0xAA
Expected DUT Response: 0xCC

Test Steps:
1. Robot sends UART_SEND 55 AA
2. Testbench sends bytes to DUT
3. DUT validates frame
4. DUT replies 0xCC
5. Robot expects 0xCC

Good presentation point: Saleae can decode the exact UART frame and prove the hardware communication.

UART Negative Test: Invalid Frame NACK

Story: The DUT should reject wrong frames, not behave randomly.

Invalid Frame: 0x55 0xAB
Expected DUT Response: 0xEE

Robot checks that DUT rejects the command.

Good presentation point: Negative tests show real test engineering, not only happy-path demos.

PWM Test Case: Motor Speed Command

Story: The DUT receives a command and changes PWM duty cycle to represent motor speed.

Command: Set speed to 50%
Expected PWM:
Frequency = 10 kHz
Duty Cycle = 50%

Good presentation point: Oscilloscope screenshot proves real electrical output.

ADC/DAC Test Case: Sensor Threshold

Story: ESP32 testbench simulates an analog sensor using DAC. STM32 DUT reads it with ADC.

Input Voltage = 1.0 V
Expected DUT State = NORMAL

Input Voltage = 2.5 V
Expected DUT State = WARNING

Input Voltage = 3.0 V
Expected DUT State = FAULT

Good presentation point: This is close to real HIL because the testbench simulates a sensor.

SPI Test Case: Register Read

Story: The testbench reads a virtual register from the DUT over SPI.

SPI Command: READ_REG 0x10
Expected Response: 0xAB

Negative Test:
READ_REG 0x99
Expected Response: Error code 0xFF

Good presentation point: Saleae SPI decode shows MOSI, MISO, SCLK, and CS behavior.

9. Feature-by-Feature Detailed Tasks

9.1 GPIO Feature

Task Description Done When
Robot GPIO keywords Create Set GPIO and Read GPIO keywords. Robot can send GPIO commands through Python.
Testbench GPIO service Implement GPIO set/read commands in Zephyr. Serial terminal can control pins manually.
DUT GPIO behavior DUT reacts to input and changes output. Input HIGH causes expected output HIGH.
Test report Run positive and negative GPIO tests. Robot report contains PASS/FAIL details.

9.2 UART Feature

Task Description Done When
UART command from Robot Robot asks testbench to send specific bytes. UART_SEND 55 AA works.
DUT UART parser DUT checks valid and invalid frames. Valid frame returns ACK, invalid frame returns NACK.
Timeout handling Robot fails if DUT does not reply in time. Timeout test produces clear failure.
Saleae capture Capture TX/RX and decode UART. Screenshot saved in repo.

9.3 PWM Feature

Task Description Done When
PWM generation/check DUT or testbench generates PWM depending on test story. 10 kHz 50% duty appears on oscilloscope.
PWM command mapping UART command changes DUT PWM duty cycle. Command 25%, 50%, 75% produces correct waveform.
Boundary tests Test 0%, 100%, invalid duty >100%. Invalid values are rejected safely.
Scope evidence Measure frequency and duty. Images saved in repo.

9.4 ADC/DAC Feature

Task Description Done When
ESP32 DAC service ESP32 generates analog voltage levels. DAC_SET CH1 1500 produces about 1.5 V.
STM32 ADC read DUT reads analog voltage and maps it to states. ADC readings are stable within tolerance.
Tolerance-based tests Robot checks ranges, not exact values. Example: 1500 mV accepted between 1450 and 1550 mV.
Analog validation Use oscilloscope/multimeter to verify DAC voltage. Evidence saved in captures folder.

9.5 SPI Feature

Task Description Done When
SPI wiring Connect MOSI, MISO, SCLK, CS, and GND. Electrical connections verified.
DUT virtual registers DUT exposes simple fake registers over SPI. Register 0x10 returns 0xAB.
Robot SPI tests Robot sends SPI read/write commands via testbench. Valid and invalid register tests run.
Saleae SPI decode Decode SPI transaction. Capture shows command and response bytes.

10. Temporary STM32 Loopback Tasks Before ESP32

Use your current STM32 board to reduce risk before the ESP32 arrives.

Loopback Test Wiring What You Learn
GPIO Loopback Output pin connected to input pin. Robot, Python, serial protocol, Zephyr GPIO.
UART Loopback UART TX connected to UART RX. Zephyr UART API, frame sending, timeout handling.
PWM Observation PWM pin connected to oscilloscope. Zephyr PWM API and scope measurement.
SPI Loopback MOSI connected to MISO, or use internal test if available. SPI transfer basics and Saleae decoding.
ADC Temporary Test Use potentiometer or voltage divider if available. Zephyr ADC API and tolerance testing.

12. Presentation Story

Your presentation should not be just a list of drivers. Tell it as a testing story:

  1. Manual embedded testing is slow and error-prone.
  2. We built a mini HIL testbench to automate embedded validation.
  3. Robot Framework sends high-level test cases.
  4. Python converts test steps into serial commands.
  5. ESP32 testbench generates real electrical signals.
  6. STM32 DUT reacts as an ECU-like system.
  7. Robot compares actual vs expected results.
  8. Oscilloscope and Saleae prove that the hardware signals are real.
Best demo: Run one Robot command live, show the DUT reaction, show PASS report, then show Saleae/scope evidence.

13. Recommended First 7 Days

Day Person A Person B Shared Output
Day 1 Create GitHub repo and structure. Setup Zephyr and flash STM32 hello world. Repo + first firmware build.
Day 2 Robot Framework + Python serial setup. Zephyr UART console / serial receive. PING -> OK PONG.
Day 3 Command parser design. GPIO Zephyr service. GPIO_SET and GPIO_READ manually.
Day 4 Robot GPIO test. GPIO loopback wiring and validation. First Robot PASS report.
Day 5 UART feature owner starts UART test. Reviewer tests GPIO and updates docs. UART loopback manual test.
Day 6 UART Robot test. Saleae UART capture. UART test + capture.
Day 7 Cleanup code and README. Prepare demo video or screenshots. Milestone 1 release.

14. Final Success Criteria

  • Robot Framework can run all tests from one command.
  • Every test has clear expected and actual results.
  • GPIO, UART, PWM, ADC/DAC, and SPI are demonstrated.
  • At least one negative test exists for UART, SPI, and ADC threshold.
  • Oscilloscope proves analog/PWM behavior.
  • Saleae proves UART/SPI frames.
  • Both team members can explain the full flow from Robot to hardware.
  • GitHub repo is clean, documented, and easy to run.

12. Presentation Story

Your presentation should not be just a list of drivers. Tell it as a testing story:

  1. Manual embedded testing is slow and error-prone.
  2. We built a mini HIL testbench to automate embedded validation.
  3. Robot Framework sends high-level test cases.
  4. Python converts test steps into serial commands.
  5. ESP32 testbench generates real electrical signals.
  6. STM32 DUT reacts as an ECU-like system.
  7. Robot compares actual vs expected results.
  8. Oscilloscope and Saleae prove that the hardware signals are real.
Best demo: Run one Robot command live, show the DUT reaction, show PASS report, then show Saleae/scope evidence.

14. Final Success Criteria

  • Robot Framework can run all tests from one command.
  • Every test has clear expected and actual results.
  • GPIO, UART, PWM, ADC/DAC, and SPI are demonstrated.
  • At least one negative test exists for UART, SPI, and ADC threshold.
  • Oscilloscope proves analog/PWM behavior.
  • Saleae proves UART/SPI frames.
  • Both team members can explain the full flow from Robot to hardware.
  • GitHub repo is clean, documented, and easy to run.
